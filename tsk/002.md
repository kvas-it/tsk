# Figure out the tooling story

tsk is a format — tools are implementations. But what operations actually
need tooling vs. just editing files by hand?

## Decision

Two complementary tools:

### `tsk` Rust CLI — for humans

Single binary, no runtime deps. Initial commands:
- `tsk new` — scan for next number, create file with front matter, open
  `$EDITOR`
- `tsk list` — formatted listing with ticket numbers, titles, statuses
- `tsk status NNN <status>` — change ticket status without opening the file

`tsk comment` deferred — you need an editor for the body anyway.

### Claude Code skill — for AI

A skill file that describes the tsk format and conventions. No binary,
just instructions. Claude reads/writes files directly — it doesn't need
a CLI wrapper.

## Why Rust?

- Single binary, no runtime — matches the "just files" philosophy
- Cross-platform without dragging in a language runtime
- Fast and snappy CLI
- Shell scripts break on YAML parsing; Python needs dependencies and
  virtualenvs
- The commands are simple enough that Rust isn't overkill

## Why a separate skill?

- AI agents don't need CLI wrappers — they can read/write files directly
- A skill is just a prompt file — self-contained, no platform dependency
- Both the CLI and the skill operate on the same files, same format
