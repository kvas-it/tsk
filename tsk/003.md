# Define front matter schema

## Resolution

Core fields (in the spec):
- `status` — string, defaults to `open` when omitted
- `created` — date (YYYY-MM-DD), just date, no time

Everything else is a custom field. The spec doesn't distinguish between
"standard optional" and "custom" — if a tool understands `tags`,
`assignee`, `priority`, or `parent`, great. If not, it preserves them.

Resolved open questions:
- Status defaults to `open` — yes, codified in spec and CLI
- Priority scheme — custom field, not in core spec. Use whatever works
- Type field — use tags. No need for a separate `type` field
- Date vs datetime — just date. Simpler, and ticket ordering within a
  day isn't important enough to add complexity
- Core vs custom — only `status` and `created` are core. Everything
  else is custom. This keeps the spec minimal and extensible
